Ahead of the Pipeline-I : A Deep Dive into Prefetching Techniques for CPU

2026-03-25

Your CPU can execute a billion instructions per second. Then it touches memory. And waits.

Before anything else: this post covers CPU prefetching only. The GPU/CUDA post briefly referenced memory prefetching and promised a deeper treatment and that part is coming in Part II. Part II will cover memory coalescing, shared memory as a programmer-managed cache, warp scheduling as the GPU’s mechanism for latency hiding, and how that whole system connects. This post is entirely about what happens on the CPU side.

The Memory Wall

Here’s a number worth sitting with: a modern CPU core can retire an instruction roughly every cycle. At 4 GHz, that’s one instruction every 250 picoseconds. A load that hits the L1 cache resolves in about 4 cycles which is fast enough to barely interrupt the flow. A load that misses all the way to DRAM takes roughly 300 cycles ….. that’s 75 nanoseconds of complete silence in a machine designed to do something useful every quarter-nanosecond.

The gap between compute speed and memory speed has been widening for decades. Processor frequency and IPC climbed steadily through the 80s, 90s, and 2000s. DRAM latency improved too, but far more slowly. The result is the memory wall: a growing chasm where your processor sits idle waiting for data that could have been there already, if only someone had thought to ask for it sooner.


The latency ladder. Each level down is a step change in wait time. The L3→DRAM gap is where performance goes to die.

Caches exist to exploit locality —> the observation that programs tend to reuse recently accessed data (temporal locality) and tend to access memory sequentially (spatial locality). They help enormously. But they’re reactive. They keep data that was already requested. Prefetching is the proactive half of the story: fetching data before the CPU asks for it, so that when the demand access arrives, the data is already there.

The Cache: What It Is Physically

Before prefetching makes sense you need a clear picture of what it’s feeding into, because the physical structure of a cache shapes everything about how prefetchers work.

A cache is built from SRAM — six-transistor cells that are fast, low-latency, and expensive per bit. They’re organised into cache lines, the atomic unit of transfer: 64 bytes on all modern x86 and ARM processors. You don’t fetch a single byte from memory. You fetch the 64-byte cache line that contains it.

The cache is set-associative. The full address space is divided into sets, and each set holds a fixed number of ways (copies). An 8-way 32KB L1 cache has 64 sets, each holding 8 possible lines. When you look up an address, you compute which set it maps to, then check all 8 ways in parallel for a tag match. A hit is fast because it’s a parallel lookup, not a linear search. A miss means none of the ways matched and you have to go to the next level.


A 4-way set-associative cache. The address is split into offset (selects bytes within a line), index (selects the set), and tag (disambiguates which line within the set). A hit requires a tag match in any way.

Replacement happens when all ways in a set are occupied and a new line needs to come in. The dominant policy is LRU (Least Recently Used): evict the way that was accessed furthest in the past. In practice, large caches often use PLRU (Pseudo-LRU) — a tree of bits that approximates LRU at much lower hardware cost.

The latency ladder in concrete terms: L1 hit ≈ 4 cycles. L2 hit ≈ 12 cycles. L3 hit ≈ 40 cycles. DRAM ≈ 200–300 cycles. Every level you fall through, the penalty roughly triples or quadruples. Miss all the way to DRAM and you’ve given up 75–100 nanoseconds that the CPU spent doing nothing.

The Bubble Problem

When a load misses the L1 cache, the instruction sits in the Reorder Buffer (ROB) waiting for its data. The out-of-order engine looks for other independent instructions to issue in the meantime that’s the whole point of OoO execution. But the ROB is finite (typically 200–500 entries on modern cores). If the miss takes 300 cycles to resolve and the ROB only has room for 300 instructions, the processor has to keep issuing those independent instructions fast enough to fill the window, or it stalls.

For irregular workloads — pointer-chasing, graph traversal, hash table lookups — the ROB runs dry. There aren’t enough independent instructions downstream of the stall to keep the execution units fed. The pipeline drains and the processor waits. This is the memory bubble: a hole in the instruction stream caused by a high-latency memory access that the OoO engine couldn’t hide.


The bubble. Once the ROB runs out of independent instructions to issue, execution stalls. Prefetching’s job is to make this bubble not happen — by getting the data there before anyone asks for it.

This is the fundamental motivation. Prefetching doesn’t make memory faster. It moves the latency to a time when nobody’s waiting on it.

What a Prefetcher Is

A prefetcher is a hardware unit wired close to the cache controller — L1 or L2, depending on the design that observes the stream of cache accesses going by and tries to predict which addresses will be needed next. When it’s confident enough in a prediction, it issues a speculative prefetch request: a memory access that isn’t the CPU asking for data, but the prefetcher betting that the CPU is about to.

The simplest way to think about it: the cache is reactive (it responds to demand), the prefetcher is proactive (it acts on prediction). Together they try to ensure that by the time a demand access arrives, the data has already been fetched and is sitting in the cache waiting.

What makes this hard is that the prefetcher has no access to source code, no knowledge of future instructions beyond what it can observe in the address stream, and has to make decisions in real time with bounded hardware state. It’s a probabilistic machine built on top of deterministic silicon, making guesses about deterministic program behaviour.

How Prefetchers Detect Patterns

The quality of a prefetcher comes down entirely to the quality of its pattern detection. Given the history of addresses accessed, what address comes next? Different prefetchers answer this differently — and the progression from simple to complex is worth tracing.

Next-Line Prefetching

The simplest possible bet: whenever you access cache line N, immediately prefetch cache line N+1. No tables, no state, almost no hardware. It works well for sequential access patterns iterating over an array, executing a linear instruction stream which are actually quite common. It’s completely wrong for everything else. Most real prefetchers include next-line as one baseline component and layer considerably more sophistication on top.

Stream Detection

A stream prefetcher watches for sequential runs of accesses — two or three consecutive cache lines accessed in order and declares a stream. Once a stream is active, it issues requests for the next N lines ahead (N is the prefetch degree) at a distance D lines ahead of the current demand access (D is the prefetch distance).

The stream detector runs as a state machine:


Stream detector states. The Candidate state is deliberate — the detector has seen one sequential miss but waits for a second before committing to a stream. One coincidental sequential access isn’t enough to start firing prefetches.

After one sequential miss, the detector moves to Stream Candidate — it saw something but isn’t ready to prefetch yet. After a second sequential miss confirming the pattern, it moves to Active Stream and starts issuing. When the access pattern breaks an out-of-order access, a different stride, a long gap it closes the stream. Most implementations run a pool of 4–8 stream detectors in parallel, each tracking a different active stream.

Stride Detection: The Reference Prediction Table

Stream detection only handles unit-stride sequential access. Many real programs step through memory with a regular but non-unit stride iterating over a struct-of-arrays with a stride of 32 bytes, stepping through a matrix column-by-column, or processing interleaved records. For these patterns, the workhorse is the stride prefetcher, built around a Reference Prediction Table (RPT).

The RPT is indexed by the program counter of the load instruction. Each entry stores: the PC tag, the previous address that instruction accessed, the observed stride (current address minus previous address), and a state field.


The RPT state machine per entry. Prefetches only fire in Steady state — after the same stride has been confirmed twice. This is the confidence gate.

Initial: just allocated, no stride observed yet. No prefetch.

Transient: one stride observed and recorded. Not trusted yet. No prefetch.

Steady: the same stride confirmed on the next access. The prefetcher fires — it issues a prefetch for current_address + stride and continues as long as the stride stays consistent.

No-pred: the stride changed from what was stored. The predictor backs off and waits to see a new stable pattern before firing again.

The two-observation gate is deliberate — it prevents the prefetcher from thrashing on a single coincidental stride match.

Confidence Counters

Both stream and stride detectors gate their prefetch decisions through saturation counters, typically 2-bit. A correct prediction (the prefetched line was actually demanded) increments the counter. A wrong prediction decrements it. Prefetches only fire when the counter is above a threshold. This is what gives the prefetcher its self-throttling behaviour — when predictions stop working, the counter drains and the prefetcher goes quiet automatically.

GHB: Global History Buffer

Stride detection fails on irregular but correlated patterns — pointer chasing through a linked list, hash table lookups, graph traversal. The addresses don’t follow a regular stride, but they do have structure: access to address A tends to be followed by access to address B, and that relationship repeats across multiple traversals of the same data structure.

The Global History Buffer handles this. It’s a circular buffer recording every cache miss address in order. A separate PC-indexed table points each load instruction to its most recent entry in the buffer. When that instruction generates another miss, the prefetcher chains through the historical entries for that PC, reconstructing the sequence of addresses the instruction has accessed before, and predicts the next one will follow the same sequence.


The GHB. Each PC’s entry in the index table points to its most recent miss in the circular buffer. Older entries for the same PC are chained together, reconstructing the access history.

GHB handles irregular correlated patterns that completely defeat stream and stride detectors. It costs more hardware state and more complex lookup logic, but it’s the right tool when access patterns are too irregular for stride detection but still have enough structure to be learnable.

The Physical Machinery

Now that we understand how the prefetcher decides what to fetch, here’s the hardware that actually executes that decision — the path a prefetch request takes from the moment it’s generated to the moment the data lands somewhere the CPU can use it.

Detector Tables

The pattern detection logic above lives as small, fast SRAM lookup tables wired to the cache controller. The RPT is one such table, indexed by PC. Stream detector state is another. These tables are read and updated on every cache access — hits and misses alike — to keep the pattern state current. A few hundred entries each is typical, sized to cover the active load instructions without growing too expensive.

Prefetch Request Queue (PRQ)

When the detector logic fires a prefetch, the generated address goes into the Prefetch Request Queue. The PRQ buffers the rate mismatch between how fast the prefetcher generates requests and how fast the memory subsystem can service them. When the memory bus is busy with demand traffic, prefetch requests sit here and wait. They’re lower priority than demand accesses — a stalled load blocking a pipeline instruction goes to the front; speculative prefetches wait.

Miss Status Holding Registers (MSHRs)

When a prefetch request leaves the PRQ and goes to lower memory, it needs to be tracked while in-flight. That tracking happens in the MSHRs.

Each MSHR entry tracks one outstanding miss — demand or prefetch. It holds: the physical address of the cache line being fetched, a valid bit marking the entry active, a pointer to the fill buffer where data should land on return, a linked list of any instructions waiting on this data (for demand misses), and subword enables marking which bytes within the line are actually needed.


One MSHR entry. The dependent instruction list allows the OoO engine to wake up waiting instructions the moment data returns — without polling.

The critical number: a typical L1 cache has 8 to 20 MSHRs. When a new miss arrives and all entries are occupied, the processor cannot issue any more loads — it fully stalls. Not because memory is slow, but because the bookkeeping hardware ran out of tracking slots. This is MSHR exhaustion.

The direct connection to prefetching: prefetch requests consume MSHRs too. An over-aggressive prefetcher fills the table with speculative entries, leaving no slots for demand loads. The prefetcher ends up stalling the very pipeline it was trying to help. This is one of the most concrete mechanisms by which too much prefetching actively hurts performance.

The Prefetch Buffer

When data returns from lower memory in response to a prefetch, it doesn’t go directly into L1 in many designs. Instead it lands in a dedicated prefetch buffer — a small SRAM structure, typically 8–16 entries, sitting between L1 and L2.


The prefetch buffer sits between L1 and L2. Speculative lines land here, not directly in L1. A demand hit promotes the line into L1. A line that’s never demanded is simply dropped.

When a demand access arrives and its line is in the prefetch buffer, it gets promoted into L1. Lines that are never demanded get dropped when newer prefetches need the space. This design keeps speculative data out of L1 entirely until it’s confirmed useful … which is the primary architectural answer to cache pollution.

So the full path is: detector fires → address enters PRQ → drains into MSHR → MSHR tracks the in-flight request → data returns from memory → lands in prefetch buffer → demand access promotes it into L1.

Prefetchers at Every Level of the Hierarchy

Prefetching doesn’t happen at just one level. There are distinct prefetchers at each level of the memory hierarchy, each under different constraints and with different degrees of sophistication.

L1 Prefetcher

The L1 prefetcher has the tightest timing requirement. If it issues a prefetch that takes 12 cycles to return from L2, it needs to have done so 12 cycles before the demand access. At 4 GHz executing one instruction per cycle, that’s 12 instructions of lookahead — a thin margin. As a result, L1 prefetchers are typically simple: next-line and a small stream detector with 2–4 active streams. Complex table lookups are too slow to be useful at this level. Accuracy matters most here because MSHR exhaustion from bad prefetches directly stalls the core.

L2 Prefetcher

The L2 prefetcher has more time — a miss to L3 takes ~40 cycles, to DRAM ~300. Both are long enough to afford more complex logic: larger RPT tables, more stream detectors, GHB-based prediction. L2 prefetchers are where most of the architectural sophistication in real chips lives. They also operate on physical addresses, which has implications discussed later.

L3 Prefetcher

The L3 is shared across all cores. A bad prefetch here can evict data another core was actively using — the blast radius is larger. L3 prefetchers are more conservative: lower aggression, stricter accuracy requirements. Their primary target is DRAM latency, which at 300 cycles gives enough lookahead time to be useful even with slower, more careful prediction.

DRAM: The Row Buffer

This level is often overlooked but it’s the most fundamental. DRAM is organised into banks, each containing a row buffer — a sense amplifier array that holds one full row (typically 8KB) after it’s been opened. Reading from an open row (a row buffer hit) takes roughly 15 ns. Reading from a closed row (a row buffer miss, requiring the current row to close and a new one to activate) takes ~60 ns.


The row buffer is DRAM’s built-in implicit prefetch. Opening a row loads 8KB into the sense amplifiers. Sequential access stays in the open row; random access keeps forcing row close/open cycles and pays 4× the penalty.

The row buffer is DRAM’s implicit prefetcher. The act of opening a row speculatively loads 8KB into the sense amplifiers. Sequential patterns exploit this naturally; random patterns that scatter across rows pay the full row-open penalty on every access. This is the physical reason sequential memory access is dramatically faster than random access even past the caches — the phenomenon goes all the way down to the DRAM silicon, and it’s worth internalising.

The memory controller also runs its own prefetch logic — reordering read requests to maximise row buffer hits and issuing read-ahead requests based on detected sequential streams.

Real chip numbers: Zen 4 runs 4 stream prefetchers at L1, 4 at L2, and a dedicated prefetcher in the memory controller. Intel’s Golden Cove (Alder Lake P-core) runs a next-line, stride, and stream prefetcher simultaneously at L2, all competing for MSHR slots, with hardware arbitrating between their outputs.

The Timing Problem

Everything so far describes what the prefetcher fetches. The harder question is when.

A prefetch is only useful if it arrives before the CPU needs the data, but not so early that the data gets evicted before the demand access occurs. There is a narrow window of usefulness, and most prefetch designs miss it in one direction or the other.


Three outcomes. Too late: the demand miss still stalls the pipeline. On time: latency fully hidden. Too early: data evicted before demand arrives — you paid the bandwidth cost and still got a miss.

Too late: The prefetch was issued too close to the demand access. Data is still in transit when the CPU asks for it. Pipeline stalls exactly as it would without prefetching. Bandwidth wasted, latency not hidden.

On time: The prefetch was issued far enough ahead that data arrived and is sitting in the cache when demand comes. Latency fully hidden. This is the goal.

Too early: The prefetch was issued so far ahead that by the time demand arrives, other traffic has evicted the line. The demand misses and goes back to memory. You paid the bandwidth cost, wasted the cache slot, and still paid the full miss penalty. Worse than nothing.

Prefetch distance controls how far ahead the prefetcher looks. Too small → always late. Too large → data evicted before use. The right distance depends on memory latency, access rate, and cache pressure — all of which vary per workload and change across program phases. No static distance is universally right, which is why modern prefetchers tune distance dynamically.

Aggression, Accuracy, and Coverage

Two metrics define prefetcher quality, and they pull in opposite directions.

Coverage is the fraction of demand cache misses that were converted into prefetch hits. High coverage means the prefetcher is catching most misses before they stall the pipeline. Coverage increases with aggression.

Accuracy is the fraction of issued prefetches that were actually demanded. Low accuracy means wasted bandwidth, evicted cache lines, occupied MSHRs — for nothing. Accuracy decreases with aggression.


The accuracy-coverage curve. Increasing aggression gains coverage but costs accuracy. The optimal point is the knee — maximum coverage before accuracy collapses. Modern prefetchers try to find and hold this point dynamically.

A prefetcher that fires on every access gets near-100% coverage but near-0% accuracy. One that never fires has 100% accuracy and 0% coverage. The hardware has to find the knee and stay there dynamically across changing workload phases.

Beyond accuracy and coverage there’s a third constraint that often becomes binding first: memory bandwidth. Every prefetch that misses the cache goes to DRAM and consumes bandwidth on a shared bus. An over-aggressive prefetcher congests the bus with speculative traffic, raising latency for demand misses even as the prefetch hit rate climbs. In bandwidth-sensitive workloads this is what you hit first.

Prefetch degree (how many lines per trigger) and prefetch distance (how far ahead) are the two main aggression knobs. Modern prefetchers tune both dynamically.

Cache Pollution and How Data Exits

Cache pollution is what happens when the prefetcher gets the pattern wrong, or fires too early. Speculated lines fill cache ways, evicting hot data the CPU was actually using. Miss rate climbs, performance degrades, and the prefetcher has made things worse.

The prefetch buffer is the first line of defence — by holding speculated data separately from L1, it prevents speculative eviction of hot L1 lines entirely. Only confirmed demand hits promote data into L1.

For lines that do enter L1 via prefetch, the LRU insertion policy determines how quickly they can be evicted. By default, new lines enter at the MRU (most recently used) position and age slowly toward LRU — occupying a cache way for a long time. A prefetched line that’s never demanded will sit there blocking a way that could hold something useful.

The fix: insert prefetched lines at the LRU position instead of MRU. They’re the first to be evicted when the set fills up. A demand hit on the line promotes it to MRU. A line that’s never demanded gets evicted quickly without displacing anything important.

BIP (Bimodal Insertion Policy) and SHIP (Signature-based Hit Prediction) go further — tracking historical reuse per line signature and adjusting insertion position accordingly. Lines that historically get reused are inserted high on the LRU stack; lines that historically get evicted unused are inserted at LRU from the start.

Feedback-driven throttling is the dynamic counterpart: the prefetcher monitors its own accuracy over a sliding window. If accuracy drops below a threshold — too many prefetched lines evicted unused — it automatically reduces prefetch degree. Intel’s prefetcher from Nehalem onward uses a variant of this. Accuracy recovers, pollution drops, and the prefetcher ramps back up when conditions improve.

So the exit story for a prefetched line is one of three things: it gets demanded and promoted to MRU (success), it ages to LRU and gets evicted to make room for something else (quiet failure), or the prefetcher throttles before it even generates the request (self-correcting).

A Taxonomy of Prefetchers

The prefetchers covered in depth above are the core mechanisms. The field is considerably wider — here’s the full landscape across hardware, software, and thread-level approaches, with links for wherever you want to go deeper.


The full prefetcher taxonomy. Each category lives at a different point on the programmer-involvement vs hardware-complexity spectrum.

Hardware Prefetchers

These run transparently — no programmer involvement, no source code changes required.


Coverage map. Sequential patterns → next-line and stream. Regular non-unit strides → RPT. Correlated irregular patterns → GHB and spatial prefetchers. The progression is increasing pattern complexity.

Next-line: Always fetch N+1 on access to N. Near-zero hardware, works for sequential access, useless otherwise. Present in almost every design as a baseline component.

Stream prefetcher: Detects sequential runs, opens a stream, issues prefetches D lines ahead at degree N. The workhorse of hardware prefetching — most CPUs run 4–8 simultaneous stream detectors.
Good starting point: Onur Mutlu’s CMU 18-740 prefetching lecture slides

Stride prefetcher (RPT): PC-indexed table, four-state machine, fires on consistent stride confirmation. Handles non-unit strides that stream detectors miss.
Baer & Chen (1991)

GHB (Global History Buffer): Circular miss buffer with PC-indexed chaining. Handles irregular but correlated patterns — pointer chasing, graph traversal.
Chen & Baer (1995)

Markov prefetcher: Builds a transition probability table from observed miss sequences. Given a miss at address A, looks up A’s most likely next address and prefetches it. Probabilistic lookahead for irregular patterns.
Palacharla & Kessler (1994)

SMS (Spatial Memory Streaming): Operates at 2KB spatial region granularity. Learns which cache line offsets within a region tend to be co-accessed. On first touch of a region, fires prefetches for the entire learned access pattern at once.
Wenisch et al. (2009), ISCA

AMPM (Access Map Pattern Matching): Maintains per-region bitmaps of accessed and prefetched lines. Matches the current bitmap against learned patterns to predict which lines to fetch next.
Ishii et al. (2011), ICS

SPP (Signature Path Prefetcher): Compresses recent access deltas into a hash signature via shift-and-XOR. Looks up the signature in a pattern table to predict future deltas. Handles variable-length delta sequences that fixed-stride RPT misses.
Kim et al. (2016), MICRO

BOP (Bouquet of Instruction Pointers): Maintains a scored set of candidate prefetch offsets. Issues prefetches at the highest-scoring offsets — effectively running many stride prefetchers in parallel and selecting the best ones dynamically.
Michaud (2016), ISCA

VLDP (Variable Length Delta Prefetcher): Extends stride prefetching to variable-length delta sequences. Matches the longest suffix of recent deltas against a pattern table. More expressive than fixed-stride RPT.
Kondguli & Huang (2018), ISCA

Bingo: Combines PC-localised and offset-based spatial prefetching using a joint PC + address offset key. Captures both instruction-specific and region-level spatial patterns simultaneously.
Bakhshalipour et al. (2019), HPCA

DSPatch: Two-component design — one PC-based, one offset-based — combined via a meta-predictor that selects between them based on recent accuracy. Adapts to whichever component is more useful for the current workload phase.
Bakhshalipour et al. (2019), MICRO

Pythia: RL-based prefetcher. The prefetcher is an agent — its action is whether and where to prefetch, its reward signal comes from cache hit/miss outcomes. First ML-based prefetcher to be competitive with hand-tuned heuristics on real workloads. The fact that it works demonstrates that memory access patterns have enough learnable structure for an RL agent to find without explicit pattern templates.
Bhatia et al. (2021), MICRO

Berti: Per-load stride table tracking actual byte offsets rather than cache-line-granularity deltas. More accurate on patterns with sub-cache-line structure.
2022, MICRO

Software Prefetchers

These require programmer involvement but can exploit access patterns the hardware can never independently infer.

Explicit prefetch instructions: PREFETCHT0 (into all cache levels), PREFETCHT1 (L2 and above), PREFETCHT2 (L3 only), PREFETCHNTA (non-temporal — bypasses cache entirely, useful for data read once and never reused). The programmer issues these at a distance ahead of the actual access. When software wins over hardware: when the access pattern is known statically but too irregular for hardware to detect — hash tables, tree traversals, graph algorithms. The programmer knows node->next is the next address to visit. The hardware has no idea.
Intel Optimization Manual, §7

__builtin_prefetch(): GCC/Clang intrinsic compiling to the right ISA instruction. Takes a locality hint (0 = no temporal locality, 3 = high temporal locality) mapping to PREFETCHNTA through PREFETCHT0.

Compiler auto-prefetch: Loop analysis inserts prefetch instructions automatically for counted loops with regular strides. Works well for dense linear algebra. Often the right default — the compiler understands loop bounds and can compute the correct prefetch distance statically.

Data-triggered prefetching: In pointer-chasing workloads, the programmer inserts prefetch(node->next) before processing node. The pointer value is available as soon as node loads — the programmer issues the prefetch the moment the next address is known, well before the CPU needs to follow it. Hardware cannot derive node->next without first loading node; the programmer already has that value.
Luk & Mowry (1996)

Non-temporal stores (movnt family): Write directly to memory bypassing the cache. For streaming writes — data produced once and never reread. Avoids write-allocate cache pollution from large output buffers.

JIT-inserted prefetches: Managed runtimes (JVM, .NET CLR) insert prefetch hints at JIT time based on runtime profiling. Adaptive and workload-specific without source changes.

Thread-Level Prefetchers

These exploit hardware parallelism to run ahead of the main thread and warm the cache before it arrives.

Runahead execution: When the CPU stalls on an L3 miss, instead of freezing the pipeline it enters runahead mode. It keeps executing speculatively, marks all results derived from missing data as INV (invalid), and lets load addresses flow into the memory system. Those loads generate new misses which get serviced in parallel. When the original stall resolves, the CPU rolls back, discards runahead results, and retires normally — but the cache is warm with the data those future loads will need. Prefetching by running ahead of yourself. One of the more elegant ideas in computer architecture.
Mutlu et al. (2003), ISCA

Dedicated helper threads: A spare hyperthread runs a stripped-down pre-computation ahead of the main thread — just enough to generate future load addresses. Its sole purpose is to touch memory early. Used in HPC and database workloads where the access pattern is complex enough that hardware can’t detect it but regular enough that a simplified pre-computation works.
Luk (2001)

Speculative precomputation (STP): Compiler-inserted lightweight threads for irregular patterns — pointer chasing, indirection-heavy loops — where hardware fails and writing manual helper threads is too expensive. The compiler extracts the dependency chain leading to a hard-to-predict load and generates a thread that computes only that chain, ahead of the main thread.
Collins et al. (2001), ISCA

Slipstream: A reduced copy of the program runs ahead on a spare thread, skipping work unrelated to memory accesses. It feeds the main thread prefetch addresses for hard-to-predict accesses. The main thread runs behind but always has its data ready.
Sohi et al. (2000), ISCA

The TLB Problem

Hardware prefetchers that look ahead in the address space hit a hard architectural wall at page boundaries.

A prefetcher operates on physical addresses — it needs them to issue requests to the memory hierarchy. But it observes load instructions working with virtual addresses. Translation goes through the TLB. For accesses within the current page, the TLB entry is loaded and translation is fast. But when a stream crosses into the next page, the prefetcher doesn’t know the physical address — the TLB might not have that entry, meaning a page table walk, which is itself a multi-level memory access.

Most hardware prefetchers respond by stopping at page boundaries. Rather than trigger a TLB walk and disrupt the memory subsystem, they conservatively limit prefetching to within the current physical page. Stream prefetchers that work perfectly across 4KB of sequential data go conservative the moment a stream steps across a page boundary.

Some newer designs run a concurrent TLB prefetcher to anticipate crossings, or operate in virtual address space with deferred translation. But most production prefetchers on shipping silicon throttle at page boundaries. If you’re profiling a workload and seeing unexpected stalls at exactly 64-cache-line intervals (4KB / 64B = 64 lines), this is why.

Measuring Whether It’s Working

Knowing the theory is one thing. Knowing whether your prefetcher is actually helping your specific workload is a different skill, and it requires measuring rather than assuming.

Modern CPUs expose hardware performance counters for this. On Intel: L1D.PREFETCH_MISS counts prefetch requests that missed L1 and went to L2. MEM_LOAD_RETIRED.L1_MISS counts demand loads that missed L1 — compare before and after enabling prefetching to get coverage. OFFCORE_REQUESTS.DEMAND_DATA_RD counts demand reads going off-core to L3 or DRAM. L2_RQSTS.PF_HIT and L2_RQSTS.PF_MISS give prefetch hit and miss rates at L2 — ratio gives accuracy. perf on Linux, VTune on Intel, and AMD uProf on Zen all expose these natively.

A useful sanity check: disable the hardware prefetcher entirely — on x86, IA32_MISC_ENABLE has prefetcher disable bits accessible via MSR or BIOS — and measure your workload’s miss rate. Then re-enable and compare. For sequential workloads you’ll see a large drop in demand misses. For random-access workloads you might see almost no difference, or a slight degradation from pollution and MSHR pressure. That’s the prefetcher telling you your workload doesn’t have patterns it can detect.

Putting It Together

The prefetcher isn’t one thing — it’s a stack of cooperating systems: detectors that find patterns in the access stream, a request queue that buffers predicted addresses, MSHRs that track in-flight requests, and a prefetch buffer that holds arriving data until it’s confirmed useful. At every step, confidence counters and throttling feedback keep the system from going too aggressive and becoming its own bottleneck.

What I find interesting about all of this is how much of it happens invisibly. The programmer writes a loop. The hardware watches addresses flow by, infers the stride, issues speculative requests, and fills the cache with data before any stall occurs. When it works, the pipeline never sees the memory wall. When it doesn’t — wrong pattern, page boundary, MSHR exhaustion — you get the bubble.

The fundamental difficulty is that the hardware operates under uncertainty. Probabilistic predictions about deterministic program behaviour, made in real time, with bounded state, no access to source code. Every design in the taxonomy above — SPP, Bingo, Pythia — is a different answer to the question of how much structure you can extract from the address stream within those constraints. The fact that an RL agent is now competitive with decades of hand-tuned heuristics says something interesting: the structure is there. Finding it is just a learning problem.

Part II will take this to the GPU. The latency problem is the same. The solution is architecturally completely different — no single instruction stream to follow, no PC-indexed stride tables, no runahead mode. Instead: thousands of warps, memory coalescing, shared memory as a programmer-controlled prefetch buffer, and warp scheduling as the GPU’s mechanism for hiding latency behind useful work. The concepts transfer. The hardware does not.
