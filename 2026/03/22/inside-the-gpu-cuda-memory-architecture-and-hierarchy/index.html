Inside the GPU: CUDA, Threads, and the Architecture That Makes It All Work

2026-03-22

Thousands of threads. One instruction. This is how GPUs think.

If you’ve read my previous post on tradeoffs in hardware, you already know the core intuition: CPUs optimize for latency, GPUs optimize for throughput. One fat sports car vs hundreds of bicycles. That framing is clean, but it’s also a bit unsatisfying once you start asking how. How exactly does a GPU achieve that throughput? What’s actually happening when you launch a CUDA kernel? What does “thousands of threads” even mean at the hardware level?

That’s what this post is about. We’re going inside — from the physical chip, through the memory hierarchy, all the way up to the programming model you write code against. By the end, the <<<grid, block>>> syntax shouldn’t feel like magic incantation anymore. It should feel like a direct map onto the hardware underneath.

Fair warning: this gets technical. That’s the point.

Why GPUs Exist: The One-Slide Version

I want to keep this section short because I’ve covered the motivation elsewhere, but the numbers are worth repeating once. An Intel Nehalem 4-core (the high-end server chip of its era) delivered about 0.8 GFLOPS/W. An AMD Radeon 5870 from roughly the same period delivered 14.5 GFLOPS/W. That’s an 18× difference in energy efficiency for floating-point work.

The reason isn’t that GPU engineers are smarter. It’s that they made a fundamentally different tradeoff: throw out everything that helps a single thread run fast — out-of-order execution, big branch predictors, large caches, memory prefetchers — and use that freed silicon budget to build hundreds of simple execution units instead. The CPU devotes most of its die to making one thread go as fast as possible. The GPU devotes most of its die to running thousands of threads at once.



Nehalem (top): most of the die is cache and control hardware. GTX 280 (bottom): almost everything is compute. That difference in silicon allocation is the entire story. Sources: PCPer, TrustedReviews

That bet pays off enormously for workloads with a lot of independent parallel computation — graphics, matrix multiplication, neural network inference, scientific simulation. For everything else, the CPU is still the right tool.

The GPU’s Anatomy: What’s Actually on the Die

Let’s start from the hardware and work upward, because everything in CUDA only makes sense once you know what it’s mapping onto.

A GPU is built around a collection of Streaming Multiprocessors (SMs). Each SM is the fundamental processing unit — think of it as a mini-processor that’s capable of running hundreds of threads simultaneously. The SM contains its own execution units (CUDA cores, tensor cores on modern GPUs), its own register file, its own L1 cache, its own shared memory, and its own warp schedulers. It’s largely self-contained.

All the SMs are then connected through an interconnect fabric to a shared L2 cache and finally to DRAM (HBM on data-center GPUs, GDDR on gaming cards). That’s roughly the full picture: many SMs + shared L2 + off-chip DRAM.


The simplified GPU: a collection of SMs, a shared L2, and off-chip DRAM. Almost everything interesting happens inside the SM.

To make this concrete: the NVIDIA A100 has 108 SMs, a 40 MB L2 cache, and up to 2039 GB/s of bandwidth from 80 GB of HBM2 memory. Each SM on the A100 has 64 FP32 CUDA cores, giving a total of 6,912 CUDA cores across the chip. When you hear “the A100 has 6912 CUDA cores,” that’s 108 SMs × 64 cores per SM.

Now let’s go inside a single SM.

What Lives Inside One SM

An SM is more complex than it looks from the outside. At its core you have the CUDA cores — FP32, INT32, and FP64 units that are the actual arithmetic execution units — alongside dedicated Tensor cores for matrix multiply-accumulate operations (MMA), critical for AI workloads. Each SM maintains a large register file (65,536 × 32-bit registers on most modern NVIDIA GPUs) shared across all threads running on that SM. The L1 cache and shared memory are physically the same SRAM, with the split software-configurable — on the A100 that’s up to 192 KB per SM, divided between L1 and shared memory as the programmer needs. Rounding it out are warp schedulers that select which warps execute each cycle, load/store units for memory operations to L1/shared/global memory, and Special Function Units (SFUs) for transcendental operations like sin, cos, and sqrt.


Inside one SM: multiple “processing blocks” each with their own warp scheduler and execution units. All blocks share the SM’s register file and shared memory.

The register file being so large is a deliberate choice. Unlike CPUs where you have maybe 16-32 architectural registers, a GPU SM maintains full register state for all active threads simultaneously. This is what enables fast context switching between warps — no saving/restoring state, it’s already there in the register file.

The Thread Hierarchy: Grid, Block, Warp, Thread

This is the conceptual heart of CUDA. The hierarchy has four levels, and each one maps to something real in the hardware.

Threads

A CUDA thread is the atomic unit of execution. Each thread runs the same kernel function but operates on different data — it knows its own position via threadIdx (a 3D variable: threadIdx.x, threadIdx.y, threadIdx.z).

Every thread has its own set of registers. It doesn’t share registers with any other thread. That’s the first and most private level of the memory hierarchy.

Blocks

Threads are grouped into blocks (also called thread blocks). A block can be 1D, 2D, or 3D — whatever makes sense for your data layout. The block’s dimensions are accessible as blockDim.

The critical hardware mapping: one block executes on exactly one SM. Once a block is dispatched to an SM, it stays there until it finishes. All threads in a block can communicate through shared memory and synchronize with __syncthreads(). Threads in different blocks cannot directly communicate — they have no shared on-chip memory between them.

The block size has a hard limit: 1024 threads per block. This is a hardware constraint — the SM’s warp schedulers and register file can only track so much state per block.

Warps: The Real Execution Unit

Here’s what most introductions to CUDA skip, which is why people write correct but slow code: the actual unit of execution on the hardware is not a thread, and not a block — it’s a warp.

A warp is 32 threads that execute in lockstep under a single instruction stream. This is called SIMT — Single Instruction, Multiple Threads. When the SM’s warp scheduler issues an instruction, it issues it to all 32 threads in the warp simultaneously. All 32 threads execute the same instruction, but each with its own registers and data.

This has been constant across every NVIDIA GPU generation: warp size is always 32.

When you launch a block of 256 threads, the SM automatically divides it into 8 warps of 32 threads each. You never explicitly program warps — but you’re always running in warps whether you know it or not.

Warp Divergence: When SIMT Goes Wrong

Because all 32 threads in a warp execute the same instruction, conditional branches are a problem. Consider this:

If threads 0-15 in a warp have data > 0 and threads 16-31 don’t, the warp can’t execute both paths simultaneously. What happens: the hardware serializes the two paths. First it executes path A with threads 0-15 active and threads 16-31 masked off (their results discarded). Then it executes path B with threads 16-31 active and 0-15 masked. The warp now takes twice as long.

This is warp divergence, and it’s one of the most common GPU performance killers. The fix is to structure your data so that threads within a warp take the same branch — called branch coherence. For the example above, if you sort your data so that all positive values come first, consecutive threads in a warp will take the same path.

Memory Visibility: What Can Talk to What

This is where things get subtle, and it’s genuinely important for writing correct parallel code. Not all threads can see the same data, and the rules are strict.

The Visibility Hierarchy

Think of it as concentric circles of visibility:

Register — Private to one thread. No other thread can see it. Fastest possible access (1 cycle). The compiler decides what lives in registers.

Local memory — Also private to one thread, but physically stored in DRAM (despite the misleading name). Used when a thread has too many variables to fit in registers (register spilling), or for per-thread arrays. Access latency is the same as global memory — slow.

Shared memory — Visible to all threads in the same block. Physically on-chip (same SRAM as L1 cache). Fast access (~29 cycles on A100). This is your main tool for communication between threads. Declared with __shared__.

Global memory — Visible to all threads across all blocks, and to the host CPU. Physically off-chip DRAM. Slow (~466 cycles on A100). This is where your input and output data lives.

Constant memory — Read-only, globally visible, cached. Used for small data that all threads read with the same access pattern (e.g., kernel weights, lookup tables). Hardware-cached so repeated reads are fast.

Texture memory — Read-only, globally visible, cached with spatial locality optimization. Originally for graphics texture sampling, useful for any access pattern that has 2D spatial locality.

The rule that catches people: threads in different blocks cannot communicate through shared memory. You can only do inter-block communication through global memory, and global memory has no synchronization guarantees between blocks — blocks may execute in any order. If you need block A’s results before block B runs, that’s a kernel launch boundary problem, not a synchronization problem.

The gap between shared memory (29 cycles) and global memory (466 cycles) is 16×. That’s why every high-performance CUDA kernel loads data into shared memory first, does all its computation there, and only writes back to global memory once at the end. Latency Hiding: How the GPU Actually Stays Busy

Here’s the thing about those 466-cycle global memory latencies. On a CPU, a 466-cycle memory stall means you’re basically idle for hundreds of nanoseconds, which is why CPUs invest so heavily in prefetching and out-of-order execution to hide that.

The GPU takes a completely different approach: hide latency with more warps.

Each SM can hold many warps simultaneously — on modern NVIDIA GPUs, up to 64 warps per SM (2048 threads). When the currently executing warp issues a global memory load and has to wait 466 cycles, the warp scheduler immediately switches to a different warp that’s ready to run. No stall. The first warp just waits in the background until its data comes back.

This only works if you have enough warps to keep the warp scheduler busy during those memory latency gaps. The ratio of active warps to maximum possible warps is called occupancy. High occupancy doesn’t automatically mean high performance — sometimes a few warps with lots of register use is better than many warps with few registers — but low occupancy is almost always a sign that the SM is sitting idle waiting on memory.

This is the fundamental difference from CPU threading. CPUs switch threads when they stall, but the cost of a context switch is high so you keep a small number of threads. GPUs make context switching essentially free (the register file already holds all state) and exploit this by keeping hundreds of warps resident simultaneously.

Synchronization: Making Parallel Work Correct

We’ve already seen __syncthreads() for intra-block synchronization. Let me be clear about exactly what it does and doesn’t do, because the scoping matters.

__syncthreads() is a block-level barrier. It synchronizes all threads within a single block. Every thread in the block must hit the barrier before any thread can proceed past it. It’s implemented as a hardware barrier instruction.

What it doesn’t do: it doesn’t synchronize across blocks. There is no equivalent of __syncthreads() that works across the entire grid. If you need all blocks to finish some work before proceeding, you have two options:

  1. End the kernel and launch a new one. Kernel launches are implicit global barriers — all blocks from the previous launch must complete before any block of the new launch begins.
  2. Use CUDA cooperative groups (newer API) which provides more flexible synchronization primitives including grid-level sync, but with restrictions on SM occupancy.

The reason there’s no general global barrier within a kernel: deadlock. If blocks can wait on each other but the scheduler doesn’t have free SMs to run the blocks they’re waiting on, you have a deadlock. The no-inter-block-sync rule prevents this class of bugs entirely.

The Execution Flow: From CPU Code to Running on Silicon

Let me walk through the complete sequence of what happens when your CUDA program runs, bottom to top. I want this to feel concrete, not abstract.

It starts on the CPU side. Your C++ code allocates device memory with cudaMalloc, copies input data from CPU RAM to GPU DRAM with cudaMemcpy(HostToDevice), and defines the grid and block dimensions.

The CPU-to-GPU copy over PCIe is slow relative to on-device memory bandwidth. If you’re doing this frequently for small transfers, PCIe bandwidth is your bottleneck, not GPU compute. This is one of the main reasons NVLink matters for multi-GPU work — but that’s a different post.

Once the data is on the device, the <<<grid, block>>> syntax triggers the GPU’s work scheduler, sending the grid and block dimensions to the hardware.

The kernel launch is asynchronous by default — the CPU doesn’t wait for the GPU to finish. It dispatches the work and immediately returns. You need an explicit cudaDeviceSynchronize() if you want to block until the GPU finishes.

From there the GPU’s global work scheduler — NVIDIA calls it the GigaThread Engine — receives the grid and starts dispatching thread blocks to available SMs based on resource availability: how many registers does this block need, does the SM have enough shared memory. An SM can run multiple blocks concurrently if resources allow.

Once a block lands on an SM, the SM divides it into warps of 32 and the warp schedulers take over. They issue instructions to active warps every cycle, while warps waiting on memory are simply parked until their data arrives.

Each warp then goes through the classic pipeline: fetch instruction, decode, issue to execution units. All 32 threads execute the same instruction simultaneously and results land in per-thread registers. For memory instructions, if data is in L1 the latency is ~38 cycles; L2 is ~262 cycles; a full DRAM miss is ~466 cycles, at which point the warp stalls and the scheduler switches to another ready warp.

If the kernel uses shared memory, threads cooperatively load data from global memory into shared memory (using global indices), sync with __syncthreads(), then perform fast on-chip computation from there. When computation finishes, threads write their results back to global memory.

Finally, once the CPU synchronizes with the GPU, it copies results from GPU DRAM back to CPU RAM with cudaMemcpy(DeviceToHost).

SM Internal Structure on A100

The A100’s SM is organized into four processing blocks, each with its own warp scheduler and dispatch unit. Each processing block has 16 FP32 CUDA cores, 1 FP64 unit, 1 Tensor Core, and 1 load/store unit.

So one SM = 4 processing blocks × 16 FP32 cores = 64 FP32 cores per SM.

This quadrant structure is important for warp scheduling: each processing block can issue one warp instruction per cycle, so the SM can be issuing to 4 warps simultaneously.

Tensor Cores: The AI Acceleration Unit

Tensor Cores are the reason the A100 is so dominant for AI workloads. A standard FP32 CUDA core does one multiply-add per cycle. A third-generation Tensor Core does a 4×4×4 matrix multiply-accumulate in a single operation — that’s 64 multiply-adds in one cycle, using FP16 or BF16 inputs with FP32 accumulation.

At the warp level, the operation is even larger: a wmma (warp-level matrix multiply-accumulate) instruction operates on a 16×16 matrix fragment spread across the 32 threads of a warp. The A100 can also use INT8, TF32, and FP64 formats in Tensor Cores depending on the workload requirements.

This is what cuBLAS uses under the hood for every matrix multiplication — tiling the large matrices into 16×16 fragments and dispatching Tensor Core operations across the SM array.

Memory System of the A100

The A100 uses HBM2 (High Bandwidth Memory, second generation) — a 3D-stacked memory technology where DRAM chips are stacked and connected through silicon vias directly alongside the GPU die. This is why it achieves 2,039 GB/s bandwidth while a typical GDDR6 GPU might do 600-700 GB/s. The tradeoff is cost and fixed capacity — you can’t upgrade HBM after manufacturing.

The 40 MB L2 cache on the A100 (vs 6 MB on the V100) is specifically sized to hold the working sets of transformer attention operations. This wasn’t accidental — the Ampere architecture was designed with AI workloads as a first-class target.

One thing the H100 made even clearer: the memory wall is real. H100 has 3.43× more compute than A100 but only 1.6× more memory bandwidth and the same 80 GB capacity. The compute is running away from the memory system. This is why quantization (reducing precision → reducing memory traffic) isn’t just a compression trick — it’s a direct response to this imbalance. Models are increasingly memory-bandwidth-bound, not compute-bound.

A100 H100 Ratio
FP16 TFLOPs 77.97 267.6 3.43×
Memory Capacity 80 GB 80 GB 1.0×

The implication: a model that fits in memory and has high arithmetic intensity (many FLOPs per byte loaded) will benefit a lot from H100. A model that’s memory-bandwidth-limited (attention with long sequences, for example) benefits much less than the raw FLOP numbers suggest.

Putting It Together: The Roofline View

If you want to know whether your kernel is limited by compute or memory bandwidth, the Roofline model is your tool. I covered this in detail in the tradeoffs post, but the GPU-specific version:

Plot your kernel’s arithmetic intensity (FLOPs per byte of DRAM traffic) against the hardware’s peak FLOP/s and peak bandwidth × bytes per FLOP ceilings. Where your kernel falls tells you what to optimize. Fall left of the ridge point — low arithmetic intensity — and you’re memory-bandwidth-limited: adding more CUDA cores doesn’t help, and the useful moves are optimizing memory access patterns, using shared memory to reuse data, and considering quantization. Fall right of the ridge point — high arithmetic intensity — and you’re compute-limited: optimize instruction throughput, use Tensor Cores if applicable, and work on occupancy.

For the A100:

  • Peak FP16 Tensor Core throughput: ~312 TFLOPS (sparse) / ~156 TFLOPS (dense)
  • Peak HBM2 bandwidth: ~2 TB/s
  • Ridge point: ~312 TFLOPS / 2 TB/s ≈ 156 FLOPs/byte

Dense matrix multiplication on large matrices easily exceeds 156 FLOPs/byte and sits firmly in the compute-bound regime — which is exactly why matrix multiply maps so well to GPUs and why Tensor Cores deliver their full theoretical throughput on large GEMMs. Sparse operations, attention with short sequences, and embedding lookups are memory-bandwidth-limited and see far less benefit from raw FLOP improvements.

Why the Abstraction Is So Good (and When It Leaks)

CUDA’s genius is a clean layered abstraction: you think about threads, blocks, and grids. You explicitly manage shared memory. You write the kernel once and the hardware runs it across thousands of execution units.

Most of this abstraction holds up well. You don’t need to think about which SM your block runs on. You don’t need to think about warp scheduling cycles. The compiler handles register allocation. The hardware handles cache management for L1 and L2.

But the abstraction leaks in performance-critical code, and knowing where it leaks is what separates fast CUDA from correct CUDA:

Warp divergence leaks the SIMT abstraction. You think in threads but execute in warps. If your branches aren’t warp-coherent, you’re serializing.

Memory coalescing leaks the flat-array abstraction. You think “each thread accesses its own element” but the hardware loads memory in 128-byte cache lines. If your 32 threads access 32 scattered bytes across 32 different cache lines, you’re paying for 32 memory transactions instead of 1. Access A[threadIdx.x] (consecutive), not A[threadIdx.x * stride] (strided), and you coalesce 32 accesses into 1 cache line fetch.

Shared memory bank conflicts leak the shared memory abstraction. Shared memory is physically organized into 32 banks. If multiple threads access the same bank in the same cycle, accesses serialize. Accessing smem[threadIdx.x] is perfect — 32 threads hit 32 different banks simultaneously. Accessing smem[threadIdx.x * 2] causes 2-way bank conflicts.

Occupancy and register pressure leak the “threads are free” abstraction. More threads per SM means more latency hiding — but each thread needs registers, and registers are a shared resource. A kernel that uses 64 registers per thread can only fit 1,024 threads on an SM (1024 × 64 = 65,536 registers, the SM’s total). Sometimes using fewer registers per thread (less ILP, more threads) wins over fewer threads with more ILP.

These aren’t bugs in CUDA — they’re places where the hardware reality shows through the software model. Understanding them is the whole game.

A Note on What’s Coming

There’s a thread that runs through this entire post that I haven’t pulled on directly: the gap between when you issue a memory request and when the data actually arrives. 466 cycles for global memory. The GPU’s answer to this is warp switching — keep enough warps resident so that by the time you cycle back around, the data is there. That works. But it’s a reactive strategy. You’re hiding latency by filling it with other work.

The more aggressive answer is to not wait for the latency at all — to issue the memory request before you actually need the data, so that by the time your computation reaches that point, the data is already sitting in cache. That’s prefetching, and it’s a deep rabbit hole.

There are two flavors of it. Hardware prefetching is automatic — the memory controller watches your access patterns and speculatively fetches cache lines it predicts you’ll need next. It works well for regular, strided patterns (sequential array traversal, for instance) and breaks down badly on irregular access patterns. Software prefetching is explicit — the programmer (or compiler) inserts prefetch instructions ahead in the instruction stream, manually decoupling the load from the use. In CUDA this shows up as __builtin_prefetch or async copies via cp.async in PTX, which are the foundation of techniques like double-buffering and software pipelining in high-performance kernels.

The Closing Thought

When you look at a GPU kernel from this angle — SM, warp, shared memory, latency hiding, register pressure — what strikes me is how much of it is a coherent response to one problem: how do you keep thousands of execution units busy when memory is 466 cycles away?

The thread hierarchy is the answer. Blocks give you cooperation. Warps give you efficient SIMD. The register file being enormous gives you free context switching between warps. Shared memory gives you a fast scratchpad to reuse data within a block. The warp scheduler gives you latency hiding. Every piece solves some part of the same problem.

The CUDA programming model is then a clean API over all of this — you manage the parts that matter (shared memory, grid dimensions, block size) and the hardware handles the rest. It leaks when you push it hard, but it holds for the vast majority of code.

What I find genuinely elegant here is that the leaks are predictable. Once you understand warps, you understand warp divergence. Once you understand cache lines, you understand coalescing. The abstraction doesn’t lie to you — it just doesn’t tell you everything until you go looking.

Go looking.
