Ahead of the Pipeline-II : Prefetching in GPUs — Coalescing, Shared Memory, and the Async Pipeline

2026-04-23

The CPU bets on prediction. The GPU bets on parallelism. Same problem, completely different answer.

Part I of this series covered CPU prefetching — the hardware units, the pattern detection algorithms, the timing problem, and the full taxonomy from next-line detectors up to reinforcement-learning-based prefetchers. If you haven’t read it, the short version is: a CPU prefetcher watches the stream of memory accesses coming from a single thread, detects patterns, and issues speculative requests ahead of demand. It works because a single thread tends to have detectable, regular access patterns.

GPUs don’t do this. Not in the same way. And understanding why is what this post is actually about.

The Same Problem, A Different Bet

The memory wall doesn’t care what kind of processor you are. DRAM latency on a GPU is roughly the same as on a CPU — 400 to 800 cycles, depending on the architecture and access pattern. That’s a long time to wait.

But GPUs made a different architectural bet from the start. CPUs threw silicon at prediction — big out-of-order windows, aggressive hardware prefetchers, deep branch predictors — to keep a single thread running fast. GPUs threw that silicon at parallelism instead. The reasoning: if you have thousands of threads in flight simultaneously, some of them are always ready to compute. When one thread stalls on memory, you switch to another. The stall still happens. You just don’t notice it.

This is latency hiding through parallelism, and it’s the foundational idea behind everything in this post. The GPU’s answer to the memory wall isn’t to predict and prefetch aggressively at the hardware level — it’s to keep so many threads in flight that memory latency is overlapped with useful compute by default.

That said, “just run more threads” only gets you so far. The rest of this post is about the layered system GPUs use on top of that, starting from hardware and going all the way down to what you write in your kernel.

Why Hardware Prefetchers Don’t Scale to GPUs

On a CPU, a hardware prefetcher works because it’s watching one thread. That thread accesses addresses in some pattern — sequential, strided, or something more complex — and the prefetcher can detect it. The signal-to-noise ratio is high because there’s only one source of addresses.

Now put 2048 threads on one SM, all accessing memory simultaneously. The address stream the memory controller sees is an interleaved mix from all of them. Any pattern one thread has is completely buried under the noise from the others. A hardware prefetcher trying to detect strides in that stream would be operating on chaos.

This is the core reason GPU hardware prefetchers are minimal. They exist — recent Ampere and Hopper GPUs have limited stream detection in the L2 — but they’re nowhere near as sophisticated as what you’d find in a modern CPU core, and they’re not expected to be. The architecture assumes the programmer and compiler are doing the heavy lifting.

Which means the layers that do work — the ones that make GPU memory access fast — are almost entirely programmer-visible.

Warp Scheduling: Latency Hiding as the First Layer

Before getting into anything the programmer explicitly controls, there’s one thing the hardware does automatically that deserves a careful look: warp scheduling.

A warp is 32 threads that execute in lockstep — one instruction, 32 data elements, every cycle. When a warp issues a memory instruction, it stalls. The warp scheduler doesn’t wait. It immediately switches to another warp that’s ready to execute. When the memory comes back — 400 cycles later — the original warp becomes eligible again and the scheduler picks it up.


Warp switching in action. While warp 0 waits on a memory access, warps 1 and 2 are executing. By the time the data returns, warp 0 slots back in without the SM ever going idle.

The arithmetic is worth making explicit. An A100 SM can have up to 64 warps resident simultaneously. A global memory access takes roughly 400–600 cycles. To fully hide that latency, you need enough independent warps to keep the execution units busy for those 400–600 cycles. With 4 warp schedulers per SM each issuing one instruction per cycle, you need roughly 100–150 in-flight instructions to cover a 400-cycle latency — which means around 4–5 warps worth of independent work.

In practice the target is occupancy — the ratio of active warps to the maximum the hardware supports. High occupancy gives the scheduler more choices and better latency hiding. But occupancy isn’t free. Every thread that’s resident uses registers. Every block that’s resident uses shared memory. If your kernel uses 128 registers per thread at 2048 threads per SM, you’ve consumed the entire register file and nothing else can run. This is the occupancy-resources tradeoff: you want more warps in flight, but each warp costs registers and shared memory, which limits how many can fit.

The practical implication: keeping register usage low and block sizes reasonable isn’t micro-optimisation. It’s the difference between the warp scheduler having four warps to work with or forty.

Memory Coalescing: Spatial Locality at Warp Scale

Warp scheduling hides latency when you have enough independent warps. But it doesn’t change how much bandwidth you consume. That’s where memory coalescing comes in, and it’s arguably the single most important access pattern property on a GPU.

When a warp executes a load instruction, all 32 threads issue their memory requests simultaneously. The memory controller looks at those 32 addresses and tries to serve them in as few transactions as possible. If the addresses are contiguous — thread 0 accesses byte 0, thread 1 accesses byte 4, thread 2 accesses byte 8, and so on — the controller merges all 32 requests into a single 128-byte transaction. One transaction, 32 threads satisfied. That’s coalesced access.

If the accesses are strided or random — thread 0 accesses byte 0, thread 1 accesses byte 256, thread 2 accesses byte 512 — each thread’s request hits a different cache line. You get 32 separate transactions instead of one. 32× the bandwidth. 32× the pressure on the memory controller. In the worst case, you can turn a memory-bandwidth-efficient kernel into a bandwidth-starved one just by getting the access pattern wrong.


Coalesced (top): 32 threads hit a contiguous 128-byte region — one transaction. Strided (bottom): each thread hits a different cache line — 32 transactions, 32× the bandwidth cost.

The coalescing requirement shapes how you should lay out your data. The classic example is matrices. If you store a matrix in row-major order and access it row-by-row (each thread in a warp reads a different column of the same row), accesses are contiguous and coalesce perfectly. If you access it column-by-column (each thread reads a different row, meaning their addresses are separated by the full matrix width), accesses are strided and you get terrible throughput.

This is why Structure of Arrays (SoA) outperforms Array of Structures (AoS) on GPUs for most workloads. In AoS, adjacent threads access data[i].x and data[i+1].x — separated by the full struct size. In SoA, all x values are contiguous: data.x[i], data.x[i+1], perfect for coalescing.

Coalescing is the GPU’s hardware-enforced version of spatial locality. The CPU prefetcher exploits spatial locality by automatically fetching the cache line around a requested address. The GPU memory controller exploits it by merging concurrent thread accesses into fewer transactions — but only if those accesses happen to be contiguous. Getting this right is the programmer’s responsibility.

Shared Memory: The Programmer-Managed Prefetch Buffer

Once your accesses are coalesced, the next layer is shared memory — the most important tool for latency hiding that the GPU programmer directly controls.

Shared memory is on-chip SRAM, physically the same memory bank as the L1 cache, but explicitly managed by the programmer rather than by hardware. On an A100, each SM has up to 164 KB of combined L1/shared memory, configurable between the two. Shared memory access latency is around 20–30 cycles — not as fast as registers, but orders of magnitude faster than global memory.

The key pattern is tiling. Instead of having each thread fetch its own data from global memory on demand, a whole thread block cooperates to load a tile of data from global memory into shared memory first — then all threads read from shared memory for their computation. The global memory load happens once, coalesced, for the whole tile. Every subsequent read is fast on-chip.


Tiling with a halo region. Each thread block loads its tile plus a border into shared memory cooperatively. Every thread then reads its neighbours from shared memory instead of global memory — eliminating redundant global reads.

The classic example is matrix multiplication. Naively, computing C = A × B has each thread independently loading an entire row of A and column of B from global memory — with massive redundancy, since adjacent threads load overlapping data. With shared memory tiling, each thread block loads a tile of A and a tile of B into shared memory, does the partial dot products, then loads the next tiles. The global memory traffic drops by a factor equal to the tile size.

Bank Conflicts

Shared memory is divided into 32 banks, one per lane in a warp. If all 32 threads in a warp access different banks, all accesses are served in parallel — one cycle. If two or more threads access the same bank (but different addresses within it), those accesses are serialised — a bank conflict. In the worst case, all 32 threads hit the same bank and you’re back to sequential access.


Conflict-free (top): each thread hits a different bank, all served in one cycle. Bank conflict (bottom): multiple threads hit the same bank, serialised into multiple passes.

The fix is usually padding — adding an extra element to each row of a 2D shared memory array so that the layout shifts relative to the 32-bank boundary — or swizzling, reordering the mapping between logical and physical addresses to spread accesses across banks. Both are about making sure that threads in the same warp always land on different banks.

The Async Pipeline: Overlapping Memory and Compute

Tiling with shared memory hides latency by reusing data. But loading a tile still takes time — you call __syncthreads(), wait for the tile to arrive, then start computing. The SM sits idle during the load.

The next layer eliminates that idle time: software pipelining using asynchronous memory copies.

CUDA’s cp.async instruction (introduced in Ampere, sm_80) copies data directly from global memory to shared memory without going through registers, and without stalling the warp that issued it. The warp continues executing other instructions while the copy completes in the background. You synchronise only when you actually need the data.

The canonical pattern built on top of this is double buffering: you maintain two tiles in shared memory simultaneously. While you’re computing on tile A, you’re loading tile B asynchronously. When computation on A finishes, B is (ideally) already ready. You flip the buffers and repeat. The memory latency of loading the next tile is completely hidden behind the compute of the current one.


Double buffering. Stage N compute and stage N+1 load overlap. When compute finishes, the next tile is already in shared memory. The memory stall disappears.

With deeper pipelines — three or four stages — you can tolerate even higher memory latency. CUTLASS uses multi-stage pipelines extensively. The number of stages is a tunable parameter: more stages hide more latency but consume more shared memory, which limits occupancy. The same tradeoff again.

cuda::memcpy_async and cuda::barrier

The higher-level C++ interface for this is cuda::memcpy_async with cuda::barrier for synchronisation. Same mechanism as cp.async, but with cleaner warp-group-level coordination semantics. The barrier tracks how many bytes are in flight and lets you wait on them precisely.

The key shift in thinking here: instead of “load, sync, compute”, the pattern becomes “issue load, compute something else, sync only when necessary.” This is software prefetching — you’re explicitly requesting data before you need it, in exactly the same spirit as a CPU software prefetch instruction, but with the pipeline structure making the timing disciplined.

Register Prefetching and the Tensor Core Pipeline

For Tensor Core workloads — the matrix multiply-accumulate operations at the heart of deep learning — there’s a layer below shared memory: register-level prefetching.

The ldmatrix PTX instruction loads a matrix fragment from shared memory directly into the register file in the layout expected by mma (the Tensor Core matrix multiply instruction). Because mma needs its operands in a specific register arrangement across the 32 threads of a warp, loading them manually with regular loads is error-prone and slow. ldmatrix does it in one instruction.

The pipeline for a high-performance GEMM kernel looks like this:

  1. cp.async loads the next tile from global → shared (asynchronous)
  2. ldmatrix loads the current tile from shared → registers (fast, on-chip)
  3. mma executes matrix multiply on register-resident fragments (Tensor Core)
  4. Stages 1–3 overlap across pipeline stages

The register file here is acting as the final prefetch buffer — one level closer to the execution unit than shared memory, completely hiding the shared memory access latency during compute.

This is what production GEMM kernels — cuBLAS, CUTLASS, FlashAttention — are actually doing. The performance gap between a naive matrix multiply and a cuBLAS call isn’t magic. It’s this pipeline, carefully orchestrated across all three levels: global → shared (async), shared → registers (ldmatrix), registers → Tensor Core (mma).

TMA: The Hopper Shift

Everything above is the Ampere-era story. Hopper (H100, sm_90) introduced something that changes the picture significantly: the Tensor Memory Accelerator (TMA).

TMA is a dedicated hardware engine, separate from the SM, that handles bulk asynchronous data movement between global memory and shared memory. Instead of having threads issue cp.async instructions, you describe the transfer to TMA through a tensor descriptor and let the hardware DMA engine handle it entirely.

The tensor descriptor encodes the shape, stride, and data type of the tensor in global memory, plus the destination in shared memory. You set it up once, then issue a TMA load with a single instruction. The SM doesn’t poll for completion — TMA signals a cuda::barrier when the transfer is done.

With cp.async, every thread in a warp participates in the copy — consuming instruction slots and execution bandwidth. With TMA, a single thread issues the transfer and the entire warp is free to do other work, or even sleep. The memory bandwidth comes for free in the background.


TMA decouples the SM from data movement entirely. The descriptor encodes source layout; the DMA engine handles the transfer; the SM gets a barrier signal when it’s done. No thread bandwidth consumed.

TMA also enables multicast — a single TMA load can broadcast the same data to the shared memory of multiple thread blocks in a cluster simultaneously. This is the H100’s cluster-level shared memory feature. One transfer, many recipients.

For large matrix multiplications, the bottleneck shifts: you’re no longer constrained by how fast threads can issue copy instructions. The TMA engine saturates HBM bandwidth independently of what the SM compute pipeline is doing. On large GEMMs, this is the primary reason H100 can actually get close to its theoretical peak throughput.

L2 Prefetch Hints and Cache Eviction Control

Below all the shared memory machinery, there are lower-level hints you can give the GPU’s cache hierarchy. These are PTX-level instructions that most CUDA programmers never touch, but they matter for specific workloads.

prefetch and prefetchu issue a prefetch hint for a global memory address into the L1 or L2 cache. Unlike CPUs where hardware prefetchers issue these automatically, on GPUs you do it manually — and they’re best used for irregular access patterns where the hardware can’t predict what’s coming.

More practically useful are cache eviction hints. A load instruction can carry a hint about how aggressively to cache the data:

.ca (cache at all levels) — the default. Data stays resident in both L1 and L2.

.cg (cache at global level) — bypass L1, cache only in L2. Useful for data that won’t be reused within the same SM but might be reused across SMs.

.cs (cache streaming / evict-first) — mark for early eviction. The right choice for data you’ll read exactly once and don’t want polluting the cache.

.lu (last use) — hint that this is the final access to this cache line. Free the cache entry immediately after the load.

The practical use case: streaming kernels that process large arrays sequentially and never reuse data. Using .cs or .cg instead of the default .ca prevents one-shot reads from thrashing the L1 and L2, leaving cache capacity for data that’s actually reused — shared memory tiles, frequently accessed constants.

Prefetching in Irregular Workloads

Everything above assumes a regular access pattern — data laid out in tensors, accessed in tiles, with clear spatial locality. A large class of real workloads doesn’t look like this.

Graph neural networks access node embeddings by following edges — which are essentially random indices into a large embedding table. Sparse attention (as in many transformer variants) attends to non-contiguous positions determined at runtime. Embedding lookups in recommendation systems access billions of parameters with random keys. These access patterns are fundamentally irregular. Coalescing is unlikely. Tiling doesn’t straightforwardly apply.

Sorting and reordering — if you know the set of indices to access in advance, sort them and group nearby accesses together. Not always possible at inference time, but during training or batch processing it can dramatically improve effective coalescing.

Software gather — explicitly load a tile of the embedding table into shared memory and have threads do their lookups from there. Works when a block’s accesses are concentrated enough in one region.

Warp-level gather with shuffle — use __shfl_sync to share addresses across a warp and collectively manage which thread fetches which element, so you can at least avoid redundant global loads.

UVM prefetch hints — cudaMemPrefetchAsync moves pages to the GPU proactively rather than waiting for a page fault mid-kernel. For workloads that access the same subset of a large dataset repeatedly, prefetching those pages before the kernel launches can eliminate the fault overhead entirely.

Irregular access patterns are where GPU performance is hardest to recover. The hardware has no prefetcher smart enough to help, the programmer has limited tools, and the physics of DRAM mean random access will always be expensive. The best strategies are usually about restructuring the problem to introduce more locality, rather than trying to fix the access pattern after the fact.

Unified Memory and Cross-PCIe Prefetching

When a dataset is larger than GPU memory, CUDA’s Unified Memory system allows the GPU to access CPU memory transparently — the GPU’s MMU generates a page fault, the driver migrates the page from CPU to GPU over PCIe, and execution resumes. It works, but a page fault mid-kernel is expensive: hundreds of microseconds of stall while the page is migrated.

cudaMemPrefetchAsync is the explicit way to avoid this. It tells the CUDA runtime to start migrating pages to a specific device before you need them, while other work is running:

The mechanism uses the same page migration infrastructure as demand faulting but runs asynchronously in the background. When the kernel launches, the pages are (ideally) already on the GPU and no faults occur. The PCIe transfer latency is hidden behind other work on the CPU or other GPU streams.

Access counters (available on NVLink-connected systems) take this further — hardware tracks which CPU pages a GPU is accessing and automatically issues prefetch requests for pages that appear to be needed. This is the closest GPU Unified Memory gets to a hardware prefetcher.

The honest assessment: for performance-critical code, explicit data movement (cudaMemcpy) with manually managed memory pools is still faster than Unified Memory with prefetch hints. The hints help, but they’re working around a system that was designed for programmability rather than peak performance. If you’re hitting PCIe bandwidth limits, restructuring your data pipeline so that CPU-to-GPU transfers overlap with computation is more reliable than relying on the prefetcher to time things correctly.

Putting It Together: The Layered Picture

The GPU’s answer to the memory wall isn’t one thing. It’s a stack, and each layer addresses a different part of the problem:

Layer Mechanism Who controls it What it hides
Warp scheduling Context switch on stall Hardware DRAM latency (when enough warps exist)
Coalescing Merge 32 thread requests Hardware + programmer Bandwidth waste from irregular access
Shared memory tiling On-chip data reuse Programmer Repeated global memory reads
cp.async / double buffering Overlap load with compute Programmer Tile load latency
ldmatrix + mma pipeline Register-level staging Programmer + compiler Shared memory access during compute
TMA (Hopper) Hardware DMA engine Programmer SM instruction bandwidth for copies
L2 prefetch hints Speculative cache fills Programmer Irregular access latency (partially)
cudaMemPrefetchAsync Async page migration Programmer PCIe transfer latency

The highest-performance GPU kernels — cuBLAS GEMM on H100, FlashAttention-3, CUTLASS conv kernels — are implementing most of this stack simultaneously. Warp occupancy tuned to the register budget. Coalesced global loads. Shared memory tiles with cp.async. Multi-stage pipelines. TMA for the bulk data movement. L2 hints for streaming reads.

None of it is magic. Each layer is solving a concrete, physical problem in the memory hierarchy. The CPU part of this story — hardware prefetchers watching a single thread’s access stream — simply doesn’t exist in the same form here. The GPU version of that responsibility sits with you, expressed through the access patterns you choose, the shared memory you allocate, and the async pipeline you construct.

That’s the real difference between CPU and GPU latency hiding. On a CPU, the hardware does most of it automatically. On a GPU, the hardware gives you the tools and expects you to build the pipeline yourself.
